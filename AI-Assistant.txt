# AI Assistant Note

## How AI Was Used

- AI was used to accelerate architecture framing, implementation sequencing, UI iteration, and repo cleanup.
- Core decisions were constrained to the single truth path: configuration, validated ERCOT data, analytics artifacts, then presentation.
- AI assistance was used as an implementation accelerator, not as a substitute for explicit verification.

## What AI Changed

- implemented the annual ERCOT large-load flexibility screener across the source-backed fetch, validation, analytics, and Streamlit layers
- added the load-profile and battery-duration drilldown model for `24/7 Training`, `Weekday 9-5 Inference`, `4h`, and `8h`
- built the interactive Texas map, reviewer ranking table, analyst console, and supporting diagnostics
- added tests for analytics contracts, reviewer-table formatting, and map-selection behavior
- hardened the submission contract with explicit runtime markers, launcher version guards, README cleanup, and removal of obsolete repo artifacts from the tracked submission surface

## Known Limitations

- the screener covers ERCOT hubs and load zones, not individual nodes
- the current screen is historical and annual; it does not forecast prices or model operational dispatch in production detail
- the battery model is stylized and excludes ancillary services, degradation, and infrastructure overlays
- the supported runtime is `Python 3.12.x`; `gridstatus==0.34.0` may not install cleanly on `Python 3.13`
- the Windows launchers were aligned to the Python `3.12.x` contract but were not runtime-verified in a Windows environment
- a fresh `Python 3.12` install could not be re-run in this local session because only `Python 3.13` was available on the machine
